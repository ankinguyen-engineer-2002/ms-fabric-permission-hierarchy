── Data-layer security (parallel, do not replace each other) ──

Supported permissions

Read (table/folder) ReadWrite (Preview) RLS (row filter) CLS (column list) OLS (table/schema)

Combination logic

∪Multiple roles → UNION (least-restrictive). Role A full table + Role B RLS = full access wins
∩Within 1 role: RLS ∩ CLS ∩ OLS (intersection)
ORMultiple roles with RLS: predicates joined with OR

Important notes

❌Admin/Member/Contributor bypass completely — only Viewer is enforced
⚠When enabled: DefaultReader role is auto-created. Must delete or remove ReadAll from existing users
⚠Cannot be disabled once enabled — per-item opt-in
✓Enforced uniformly: Spark, SQL endpoint, Power BI, OneLake API

T-SQL tools

GRANT / REVOKE DENY (override GRANT) CREATE SECURITY POLICY (RLS) GRANT SELECT (column list) ALTER TABLE MASKED (DDM) Views · SPs · Functions

Differences vs OneLake Security

⚡DENY overrides GRANT — opposite of OneLake (most permissive wins)
✓DDM supported — OneLake Security has no DDM
⚠Workspace Admin bypasses T-SQL DENY
✓Views/SPs can use SQL GRANT independently of table permissions

Security types

RLS (DAX row filter) OLS (hide table/column) CLS (column restrict)

Direct Lake mode gotcha

⚡Direct Lake: RLS enforced at semantic model — does not go through OneLake Security
⚠DirectLake over SQL mode: uses owner's credentials — does not pass user identity through shortcut
✓DirectLake over OneLake mode: user identity passes through correctly

Build permission risk

❌User with Build permission → connect directly to dataset → bypass report-level RLS
✓RLS at model level still applies even when building custom reports
⚠OLS hides table/column from schema — not a true security layer
